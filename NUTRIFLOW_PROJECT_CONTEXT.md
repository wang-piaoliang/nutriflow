# NutriFlow 项目上下文

> 最后更新：2026-07-24
>
> 用途：让新的 Codex 任务无需依赖旧聊天记录，也能继续维护本项目。开始工作前同时阅读根目录 `AGENTS.md`。

## 1. 项目目标

NutriFlow 是用户自用的中文手机 PWA，用来完成三件事：

1. 首页查看每天和每顿的摄入参考。
2. 食材页查看每周采购总量、食材轮换和个人优先级。
3. 饮食页查看已确认的每日餐食记录。
4. 采购页维护当前正在吃的食品、已吃完历史、逐次采购小票和本机私密照片。

饮食页和采购页都支持添加只保存在本机的私密照片。

用户会在单独的手机 ChatGPT 对话里持续发送小票和每日饮食，再把结构化同步包交给 Codex 更新本仓库。

## 2. 当前产品状态

- 主应用：`public/nutriflow.html`
- PWA：`public/manifest.webmanifest`、`public/sw.js`
- 根路径：`app/page.tsx` 和 `public/index.html` 均转到 `/nutriflow.html`
- 图标：根 `public/` 下的 `apple-touch-icon.png`、`icon-192.png`、`icon-512.png`、`maskable-512.png`
- 当前离线缓存：`nutriflow-pwa-v48`
- 应用壳更新机制（2026-07-23）：`nutriflow.html` 注册 SW 后，监听 `controllerchange`，新 SW 接管时自动 `location.reload()` 一次（用 `hadController` 跳过首次安装那次），并在 `visibilitychange → visible` 时再 `registration.update()`。这是为了解决**独立/桌面 dock app 停在旧版本**：Safari 每次导航都会重新检查 SW 所以总是最新，dock app 会常驻、只吃旧缓存壳。SW 侧 `install` 有 `skipWaiting()`、`activate` 有 `clients.claim()`，配合页面的 reload 让 dock app 冷启动或回前台时自动切到新版。
- 底部导航顺序（2026-07-24 改）：`饮食`、`采购`、`食材`、`目标`。默认落地页是 `饮食`（其 `<section>` 和第一个导航按钮带 `active`）。最后一个 `目标` 是原来的 `首页`——只改了导航文案和顺序，`data-view="home"`、`id="home"` 及页内内容都不变。
- 数据尚未拆成 JSON，食材和采购记录仍写在 `public/nutriflow.html` 的 JavaScript 数组中。
- “吃完”状态保存在当前设备和当前网址的 `localStorage`，键为 `nutriflow_consumed_v1`；它不会自动跨手机、电脑或不同域名同步。
- 2026-07-21 用户明确决定维持这一现状：**不要**为了同步而把“吃完”写进 `purchases` 数据，也不要新增 `consumedAt` 之类字段。
- 勾选“吃完”完全由用户自己在 App 上完成，不属于 Codex 的工作。收到含「库存变化 / 消耗」的同步包时，不要写入仓库、不要假装已处理，也**不要提醒或建议用户去勾选或取消勾选**——用户不需要这类操作指引。只有当数据本身出现矛盾、可能导致记录出错时，才简短说明事实。

### 首页

- 先显示每天核心总量，再显示每顿目标和完整每日目标。
- 《中国居民平衡膳食宝塔（2022）》图片放在「每天目标」卡片内部的列表下方，不再单独占一张卡片——用户要求两者合成一个 section。
- 「每天目标」每一类都带 emoji 图标（`dailyTargets[].icon`），首页顶部三块核心数值也用同一套图标。
- 鸡蛋按用户要求挪到最后一行「🥛 奶/酸奶/蛋」（`amount` 为 `300-500ml + 蛋 1 个`），「鱼禽肉蛋」行不再提蛋。
- **但克数口径没有跟着改**：膳食指南把蛋归在鱼禽肉蛋里，那 120-200g 是含蛋的。不要因为蛋显示在奶那一行，就把 120-200g 改小或给蛋另立克数——那会和指南冲突。这条口径只记在本文档里：用户嫌那行说明太长，界面上的 `note` 已精简成“🥚 鸡蛋保留蛋黄。”，不要再把“克数仍计入上面那 120-200g”之类的解释写回页面。
- 「每顿目标」四餐带图标：🍞 早餐 / 🍱 午餐 / 🍲 晚餐 / 🥜 加餐。**必须是食物图标**——先前用过 🌅🌞🌙🍏 这套日出/太阳/月亮的时间线，用户直接反馈不好看：全 App 其他地方（每天目标、食材页、采购清单）都是食物 emoji，混进风景和天体就格格不入。选的时候避开 `dailyTargets` 已占用的 🥩🥦🍚🍎🥛。
- 顶部三块核心数值在 ≤430px 时字号压到 16px。104px 宽的格子放不下 18px 的“350-500g”，会断成两行。
- 核心卡片使用暖白低饱和配色，避免大面积高饱和红绿蓝。
- 手机端正文尺寸已经压低，长内容应自然换行，禁止横向滚动。

### 食材

- 周采购分类：鱼禽瘦肉、蛋奶豆、蔬菜、主食、水果坚果。
- 动物性食物采用合并额度：鱼禽肉每周合计 600-1000g，其中水产 300-500g；蛋类每周 300-350g（约 7 个）。
- 蛋只在「蛋奶豆」这一类里出现。用户要求「鱼禽瘦肉」卡片下不再提蛋，所以该分类的 `note` 里删掉了“另加蛋类 300-350g，动物性食物总计约 900-1350g”——后半句的总量是把蛋算进去才成立的，蛋既然不在这张卡片上说，这个合计数也一并移除，避免出现没有依据的数字。营养口径本身没变，只是不在这里展示。
- 每日鱼禽肉蛋合计 120-200g。
- 水产、禽肉和红肉按分类分配频次；单项食材是替换关系，不把每张食材卡的次数相加。
- 牛肉排在瘦猪肉前；“猪小里脊/通脊”明确属于猪肉。
- 鸡腿、鸡小胸、鸡胸合并为“鸡肉”一组；不列火鸡。
- 富脂鱼优先于白肉鱼，虾贝可增加多样性但不能完全替代富脂鱼的长链 Omega-3。
- 当前蔬菜库为 16 组，水果坚果库为 11 组；苹果和香蕉均为 4.4，避免把常见完整水果不当地排成低营养价值。
- **发现用户吃了/买了食材库里没有的东西，默认就要新增条目补上**，这是用户明确定下的规则，不用再问。萝卜（白萝卜/青萝卜/樱桃萝卜，4.2 分）就是这么补进来的——采购记录里早有 `foodId:"radish"`，图标和分组规则也都在，唯独食材库缺了条目。新增时记得同时检查 `iconForFood`、`purchaseGroupRules` 和 `dietItemCategoryRules` 是否都覆盖到。
- 权威指南不会为单个食材逐一排名。页面分数仅按指南契合度、营养密度、轮换价值和个人目标做保守参考，不是通用健康分或诊断结论；0.1-0.2 分差不表示明显优劣。
- 单项卡片只显示建议频率，不再重复显示上方已经汇总过的建议份额。

### 饮食

- 饮食页按日期展示已确认的餐食记录；新收到餐食照片或文字后，写入 `dietRecords`。两条录入通道：手机 ChatGPT 对话整理成同步包，或用户直接在 Claude Code thread 里发照片（见第 5 节「直传照片」约定）。
- 目前的 `dietRecords` 仍是硬编码数组，尚未做设备间自动同步。
- 记录格式：`{date, summary, meals:[{name, items:[], note}]}`。`date` 用 `YYYY-MM-DD`，页面按它倒序排列；`summary` 省略时自动显示“N 餐”，现在所有记录都省略它。
- 同步包没给估算克数时不要编造，但**也不要在界面上写“未提供估算量”之类的说明**——用户明确要求去掉这类文字。没有克数就只列食物名，日期右侧只显示“N 餐”。
- 页面标题是「每天吃了什么」，不是「每天实际吃了什么」。说明文字（2026-07-24 改）为“每顿饭发给 Claude 识别，自动整理成记录；照片仅存本地不上传。”——用户要求简短，不要再加回“轻点看大图，长按可删除”之类的操作说明。

#### 本周吃到

- 「本周吃到」和「每天吃了什么」在**同一张卡片**里，用 `.diet-subhead`（上边距 + 分隔线）隔开，不是两张并列的 card。用户明确要求合并，不要再拆开。
- 「本周吃到」总结（2026-07-24 改版）：改成和首页顶部「今天吃到这些」同款的指标卡片——每个有记录的分类一个 `.metric` 磁贴（`grid3` 布局，大数字 + `图标 分类名`），下面一行 `.week-caption` 写“本周共 N 种食物…”。数据口径没变：同一样吃多次只算一种，用 `itemTag` 去重；一种都没吃到的分类不显示。渲染在 `renderWeekSummary`，旧的单行 `.week-line` 样式已删。
- 曾经按分类逐行展开并列出每样食物名，占掉大半屏，用户要求“写总结就行”。**不要**再改回逐类展开的形式。
- 分类复用 `dietItemCategoryRules`（含 `icon`），所以和餐次内排序用的是同一套规则，改关键词会同时影响两处。
- 一种都没吃到的分类**直接不显示**，也不提示“还没吃到”。用户的理由是一周还没过完，这时候点名缺什么言之过早。
- 周的起止用 `currentWeek()`（周一为起点），判断记录是否属于本周用 `sameWeek()`。`sameWeek` 已改为用 `localDate()` 按本地日期解析：`new Date("2026-07-20")` 是 UTC 午夜，在负时区会落到前一天，导致周一的记录被算进上一周。
- 现有记录：2026-07-24（午餐）、2026-07-23（午餐、晚餐）、2026-07-22（午餐、晚餐）、2026-07-21（午餐、晚餐）、2026-07-20（午餐、晚餐）。除 07-24 外来自同步包；07-24 由用户在 thread 直传照片录入。均只有食物名称、没有估算量。源数组按日期倒序书写，页面也按日期倒序渲染。
- 2026-07-24 午餐由用户直传单张照片（约定：1 张=上一顿）识别：牛肉、肉丸、煎蛋、一杯牛奶、整颗番茄、花菜、胡萝卜、毛豆，主食为多谷杂粮饭。饮品按 `牛奶` 记（含 `奶` 关键词，归蛋奶豆）；若实为豆浆再改。
- 主食经用户补充确认：用的是「盒马有机七色糙米」（配料：有机燕麦米/黑米/糙米/高粱米/荞麦米/大麦米/红米），用户又自己加了大米、藜麦、小米。写作 `{name:"七色糙米饭（＋大米·藜麦·小米）", as:"杂粮饭"}`——页面显示具体品名、统计按 `杂粮饭` 归主食。用户随后确认「22 号及以后的米饭都是这款」，故 07-23 午餐的主食也改成同一写法（07-22 没记米饭；晚餐是乌冬面/米线，不是米饭，不动）。07-20、07-21 的 `藜麦米饭` 保留（在 22 号之前）。07-23 和 07-24 共用同一 `杂粮饭` 标签，「本周吃到」里去重成一种。
- 2026-07-23 由用户上传的两张餐食照片识别：午餐为鸡胸肉煎排+两颗肉丸+胡萝卜+毛豆+花菜（花椰菜/西兰花混合，沿用旧日同款菜品的 `花菜` 写法）+藜麦紫米饭；晚餐为砂锅米线（煎蛋、炸豆皮、生菜、米线）。煎蛋按 `煎蛋` 写入（含 `蛋` 关键词，正确落到蛋奶豆）；`豆皮` 起初不匹配任何分类关键词、被归到未分类，已按“吃到没覆盖的就补上”的规则把 `豆皮` 加进「蛋奶豆」关键词。
- 2026-07-22 午餐的鸡蛋按 `蛋` 写入，不写成 `鸡蛋`：`dietItemCategoryRules` 里「鱼禽瘦肉」的关键词 `鸡` 排在「蛋奶豆」前面，写成 `鸡蛋` 会被 `鸡` 命中而错分到鱼禽瘦肉。`蛋` 不含 `鸡`，能正确落到蛋奶豆。以后遇到蛋一律写 `蛋`。
- 餐次内的食物按分类顺序显示：鱼禽瘦肉 → 蛋奶豆 → 蔬菜 → 主食 → 水果坚果，与「食材」页的分类顺序一致。排序在渲染时由 `sortMealItems` 完成，`dietRecords` 保留同步包上报的原始顺序，不要预先排好再写进数据。
- 分类靠 `dietItemCategoryRules` 的关键词匹配，规则按数组顺序命中，靠前优先。顺序有意义：蔬菜必须排在水果坚果之前，否则“冬瓜”会被“瓜”判成水果；“毛豆”归在蛋奶豆且排在主食之前，否则“毛豆米”会被“米”判成主食。新增关键词时要照顾这类冲突，并在测试里补一条断言。
- 无法识别的食物名排在最后，并保持它们之间的原有相对顺序，不会被丢弃。
- **食物条目可以带隐藏标签**：写成 `{name:"烤鸭", as:"鸭肉"}`，页面显示「烤鸭」，分类和「本周吃到」的去重用 `as` 的值。用户的要求是"烤鸭就直接写烤鸭，我想知道具体吃了什么菜，统计的时候可以写鸭肉"。取值一律走 `itemLabel()`（显示）和 `itemTag()`（归类/统计），不要直接用条目本身，否则遇到对象会渲染成 `[object Object]`。
- 只有当菜名本身不含任何分类关键词时才需要加 `as`（例如「罗宋汤」）。像「烤鸭」这种已经含「鸭」的，加 `as` 只是为了统计时和「鸭肉」合并成一种。
- **在外就餐（2026-07-24 约定）**：那一顿加 `place` 字段写餐厅名，页面上以 `📍 名称` 显示在该顿食物最前面（`.meal-place`，加粗），`place` 在 `allDietRecords()` 合并时会保留（和 `note` 一样）。餐厅餐的食物写简短**菜名**（如「宫保鸡丁」「罗宋汤」），别只写食材、也别写太长；菜名不含分类关键词时用 `{name, as}` 让它能归类和去重。在家做的饭照旧只列食材、不加 `place`。07-24 晚餐寿司郎已用 `place:"寿司郎"`。
- **用户可以在页面上手动补记餐食**，两个入口，都是小图标、不带文字标签（用户明确说"不用加这个字，可以直接在后面加小icon"）：
  - **每一顿后面的 `.chip-add`（虚线圆形 ＋）**——最常用的那个。点开就地展开一个输入框，往这一顿里加食物，不用回到表单重选日期和餐次。它**渲染在 `<small>` 食物列表的末尾、和文字同一行**（inline-flex + `vertical-align`），用户明确要求不要另起一行；不要把它移回下面的 `.chip-row`。删除用的胶囊（`.chip-row`）仍在下一行，只在这一顿有手动补记时才出现。
  - **「每天吃了什么」标题右侧的 `.head-add`（＋）**——只在整天或整顿还不存在时才用得上，展开 `#dietDayForm`（日期 + 餐次 + 食物）。该按钮在 `dietLogList` 外面，重渲染不会重建，所以**只绑一次**，不要挪进 `bindDietForm`，否则每次渲染都会叠加一个监听器。
- 食物输入统一走 `parseItemInput()`：空格、逗号、顿号、加号都当分隔符。
- 存在 `localStorage` 的 `nutriflow_diet_entries_v1`，默认是**设备本地**的、不进 Git。**可选云同步**（2026-07-24 加）：用户自建的 Cloudflare Worker + D1（代码在 `api/`）能把这段手动补记同步到云端、跨设备。前端在「目标」页最下有「☁️ 云同步」卡片。Worker 地址有默认值 `DEFAULT_SYNC_URL`（`https://nutriflow-sync.nutriflow.workers.dev`，用户 2026-07-24 已部署的实例；地址不是密钥、可写死），输入框预填、`syncConfig()` 也用它兜底，所以**每台设备通常只需填一次口令**（存 `nutriflow_sync_token`；地址想换就填别的、存 `nutriflow_sync_url` 覆盖默认）。填口令即启用：打开页面 `syncPull()` 拉取、每次 `saveDietEntries()` 走 `schedulePush()` 防抖 `PUT`、回前台再拉一次；`Authorization: Bearer <口令>`。**不配置就零影响**，完全按本地走。照片（IndexedDB）和"吃完"（`nutriflow_consumed_v1`）**不**走同步，仍设备本地——延续 2026-07-21 的决定。同步只覆盖 `dietEntries`，`dietRecords`（代码里的）不受影响。
- `allDietRecords()` 把 `dietRecords`（同步包写进代码的）和手动补记合并：同一天同一餐合成一条，餐次按早餐/午餐/晚餐/加餐排序。手动补的那几样额外记在 `meal.added` 里，界面上显示成可点的胶囊按钮，**只有它们能删**——同步包写进代码的删不掉，避免用户误删了下次发版又回来、造成困惑。
- 渲染饮食页和周总结都要走 `allDietRecords()`，不要直接读 `dietRecords`，否则手动补记的内容不会出现在统计里。
- 三餐的主食“藜麦米饭”最初没有出现在同步包里（照片里有）。原因是同步包模板里“跳过…米等主食”的过滤规则本意只针对采购，却被套用到了饮食上。模板已改：过滤规则明确限定为只作用于「新增采购」，饮食部分要求把所有出现的食物写全，并单列“主食”字段。
- 每天的记录下方有一块本机私密照片区：可上传多张餐食照片、显示缩略图、点击后在大图查看器中打开、可单张删除；有照片时日期右侧显示 `📷 N`。存储和隐私性质与小票照片完全相同，只保存在当前设备的浏览器里。
- `renderDietLog` 因为要读 IndexedDB 已改为 `async`，并像 `renderShopping` 一样用 `dietRenderVersion` 做并发渲染保护。`render()` 用 `void renderDietLog()` 调用它；测试里要 `await` 它，否则运行时异常会变成静默的 rejected promise。
- “虾”和“肉丸”在采购记录里没有对应条目。用户确认饮食记录照实保留、不为它们补造采购记录。饮食记录与采购记录本来就相互独立，出现这种不匹配时不要自动补数据，也不要反复追问。
- 同步包的「消耗」与页面上的“吃完”不是一个含义：消耗指当天用掉了一些，吃完指整件用尽。花菜、胡萝卜、毛豆在 07-20 和 07-21 两天的消耗列表里都出现过，说明它们当时并没有吃完。记住这个区别是为了不误读数据，但库存状态本身由用户自己维护，不要据此给出操作建议。

### 采购

- 当前示例来自 2026-07-17 的 fudi 超市五道口店小票：7 件，合计 65.73 元。
- 小票以一次采购为一个默认折叠区块；店名、时间、件数、分类重量和总价只在小票级别显示一次。
- 当前食品包括鸡小胸、梅花肉、毛豆米、西红柿、松花菜和胡萝卜；购物袋只进入采购历史，不进入待吃清单。
- 记录范围（2026-07-23 改）：**采购记录要完整**，小票上买的每一样都记，包括主食（米/面/面包）、油、盐、调料、牛奶等备货，不再只记生鲜。仍按稳定 ID 去重真正重复的行。
- **备货（pantry）与生鲜的区分**：主食/油/盐/调料等备货写入数据时带 `pantry:true`。`pantry:true` 的商品会出现在采购历史、小票分类重量汇总、食材页本周采购量、以及元/kg 单价对比里，但**不进入「现有食材」勾选清单和「已吃完历史」**——即用户不需要对它勾选是否吃完。购物袋仍用 `foodId:"bag"` 单独排除。生鲜（肉、水产、蔬菜、蛋、豆制品、水果坚果）不带 `pantry`，正常进入现有食材。逻辑在 `renderShopping` 的 `boughtItems` 过滤（`!purchase.pantry`）。
- 2026-07-18 盒马鲜生新增“国产谷饲黄牛牛腱肉”400g，24.90 元。椰子水按当前规则不记录；该单的整单总价始终未确认，页面显示的 24.90 只是已记录商品的合计。
- 2026-07-22 盒马鲜生（大钟寺店）新增一单（`receiptId` `2026-07-22-hema`），共 5 件：长白萝卜（foodId `radish`，新增图标 🥬 并加入 `purchaseGroupRules` 蔬菜组）、盒马大颗粒虾滑（foodId `shellfish`，2 袋×150g 计 300g，24.28 元）、盒马日日鲜生菜（foodId `leafy`）、鲜香菇（foodId `mushroom`），以及盒马乌冬面（foodId `noodle`，600g=200g×3，8.65 元，`pantry:true` 备货，不进现有食材）。该单整单总价同步包写 42.42 元，而 5 件明细相加为 40.56 元（差 1.86，来源不明、未擅改），故 `receiptTotalKnown:false`，页面小票总额显示 40.56 元。虾滑按 300g 计约 80.9 元/kg，是「食材单价对比」里最贵的一条；乌冬面约 14.4 元/kg 也在对比里。
- 用户不希望在小票上显示“仅记录牛腱肉；椰子水未记录，订单总价待确认”这类解释性备注，该 `receiptNote` 已移除。`receiptNote` 字段和渲染逻辑保留，除非用户明确要求，不要再给小票补写这类说明文字。
- 用户要求金额直接显示数字，**不要**再加“已记”前缀，小票级和「采购历史」标题右侧的汇总都一样。`receiptTotalKnown` 字段仍保留在数据里以记录“整单总价未确认”这个事实，但目前不在界面上体现；不要因为它没被显示就把它删掉，也不要擅自把前缀加回来。
- 勾选吃完后，食品移入“已吃完历史”；默认只显示最近 1 件，其余折叠。每件显示购买日期和吃完日期；历史布尔值因没有时间戳会标为“日期未记录（旧记录）”。
- 每个展开的小票可添加多张私密照片，缩略图可点击后在大图查看器中打开；点大图本身、黑色背景或右上角关闭按钮都可缩回。照片 Blob 仅保存在添加时所在设备、所在浏览器和同一网址的 IndexedDB（数据库 `nutriflow_private_photos_v1`）中；不写入采购数组、不提交 GitHub、不发布到 GitHub Pages，其他人和其他设备看不到。
- 私密照片能力由采购和饮食两处共用，实现集中在 `listPrivatePhotos` / `savePrivatePhoto` / `removePrivatePhoto` / `photoSectionMarkup` / `bindPhotoControls`。照片按 owner 归属：小票用 `receiptId`，饮食用 `diet:<日期>`（见 `dietPhotoOwner`）。IndexedDB 里的字段名仍叫 `receiptId`，是为了让已存在的照片记录无需迁移即可继续读取；新增其他归属时沿用同一字段，不要改 schema 或升 DB 版本。
- 对象 URL 按区隔离保存在 `activePhotoUrls = {shopping, diet}`，各自渲染时只回收自己那一组。不要改回单一数组：那会让重渲染采购页时吊销饮食页仍在显示的缩略图 URL。
- 「照片仅保存在这台设备」这句话每个板块只写一次：饮食写在页面顶部的说明框里，采购写在「采购历史」标题下的 `.private-note` 里。**不要**再把这句话塞回每条记录下面——用户明确要求不要重复。
- 添加照片的 📷 按钮：**36×36 的实线小方块，单独一行、靠右**（`.photo-tools`），缩略图网格紧跟在下面。这个位置反复调过三轮，最终定回最早那版：放进网格最后一格试过（用户明确说更喜欢在右上角），撑满整格的大方框也试过（太大）。所以要守住：**独占一行且右对齐**、**36×36**。
- 那一行和缩略图之间必须收紧（`.photo-tools` 上 `padding-top:8px`，`.photo-grid` 只有 `margin-top:6px`，分隔线画在 `.photo-tools` 顶部而不是网格顶部）。之前两者各带 12px 间距，中间看着像空了一整行，用户专门提过。没有照片时不渲染 `.photo-grid`。
- 删除照片是**长按缩略图 550ms**，然后弹确认框。缩略图上原来常驻一个 `.photo-remove` 的 ×，在手机上想放大时极容易误触，用户要求去掉，该类已删除，不要加回来。
- 长按实现的几个要点，改动时不要破坏：按住时加 `.holding` 类做缩放变暗反馈；指针移动超过 10px 就取消（那通常是在滚动页面）；`contextmenu` 要 `preventDefault`，并配合 `.photo-open` 上的 `-webkit-touch-callout:none`，否则 iOS 长按图片会弹出“存储图片”菜单；长按触发后系统仍会补一个 `click`，用 `deletedByHold` 标记挡掉，否则删完会紧接着弹出大图查看器。
- 轻点仍然是打开大图。两个板块的说明文字里都写了“轻点看大图，长按可删除”，因为长按不像常驻按钮那样一眼可见。
- 缩略图统一加了 `filter:brightness(.94) saturate(.88)` 和一层薄纱 `.photo-tile::after`。餐食照片多是白盘白米饭，直出比暖白低饱和的界面亮一大截，会盖过文字内容。大图查看器不加任何滤镜，保证看到的是原图。
- **照片入库前会做归一化**（`normalizePhotoBlob`）：画进 canvas、长边压到 1600px、导出 `image/jpeg` q0.86。这是为了解决用户反馈的"有的照片过曝、每张亮度色调都不一样"——iPhone 存的是 HDR 照片，Safari 会按扩展动态范围渲染，比普通图亮一大截，而 CSS 滤镜作用在已经被提亮的结果上，压不住。canvas 输出必然是 SDR/sRGB，HDR 增益在这一步被压平。顺带按平均亮度做一次温和的曝光归一（目标 0.56，增益夹在 0.72–1.25），让明暗不一的照片落到接近区间；夹紧是刻意的，超出范围就不再拉，避免把照片"修"得失真。
- 归一化失败时（解码不了等）原样保存，宁可亮一点也不能把照片弄丢。
- 记录上带 `normalized:true`。`loadPhotosByOwner` 会把没有这个标记的旧照片就地转换并写回 IndexedDB，所以升级前存的照片第一次打开时自动统一，不需要用户重传。新增其他归属或改存储时不要丢掉这个惰性迁移。
- 原始照片可继续由用户保存在 iCloud；PWA 不读取 iCloud 私有文件路径，也不保存 iCloud 共享链接。清除该浏览器的网站数据会同时清除本机照片附件。
- 「购买的食品」已改名为「现有食材」。这份清单会越来越长，所以条目标题压到 14px/500 字重、说明文字 12px、行内边距 10px，目的是一屏多放几件。
- 条目副行写成一行：`638g · 购于 2026-07-17 fudi 超市五道口店`，中间**不要**再插 `<br>`，太长时让它自然折行。用“购于”而不是“购买于”也是为了省一点宽度。已吃完的条目把“吃完于 …”接在同一行末尾，同样不换行。`.buy-row strong` 和 `.buy-row small` 的覆盖规则必须写在 `.item strong,.buy-row strong,.receipt-line strong` 那两条共享规则**之后**，否则同优先级下 font-size 会被共享规则盖掉。

### 食材单价对比

- 位于采购页最后一张卡片，把每次采购统一换算成元/kg 后横向对比，用的是本来就存在、之前一直没用上的 `.bars` / `.track` / `.fill` 样式。
- 只统计 `foodId !== "bag"`、按重量购买（`isWeighedAmount`）且有金额的采购。计件商品（例如“1 个”购物袋）必须排除：`parseAmount` 会把“1 个”当成 1 克，换算出来的单价是天文数字。
- 按 `foodId` 分组（没有则用商品名），同一食材的多次采购归到一组；组内按日期倒序排列，组间按最近一次的单价从贵到便宜排序。
- 条宽按**全局**最贵单价缩放，不是按组内最大值。这样不同食材之间也能直接比长短；若按组内缩放，只买过一次的便宜食材也会顶满一条，看上去像很贵。
- 一组内有多次采购时，最贵的用 `.fill.hot`（红）、最便宜的用默认绿、其余用 `.fill.warn`（黄），标题右侧显示“N 次 · 最低–最高”。只买过一次时不做红黄绿区分，只显示“1 次”。
- 计算出的元/kg 与小票上印的单价一致（例如鸡小胸 27.6、胡萝卜 4.0）。注意浮点：24.90 / 0.4 得 62.2499…，`toFixed(1)` 是 62.2 而不是 62.3，写测试时不要按手算的四舍五入去断言。

## 3. 个性化与隐私

- 公开仓库只记录实现所需的高层营养方向，不提交用户的生日、体检数值、完整诊断或医疗报告。
- 本机存在时，具体个性化依据存放在 `.nutriflow-private-context.md`；该文件已被 Git 忽略。
- 当前高层方向：规律吃够、适度增重、保证蛋白质和铁、不过度压低主食、避免依靠高油食物增重。
- 用户已明确同意公开展示采购和饮食记录。
- 即使如此，也不要公开付款信息、手机号、会员号、条码、精确住址或医疗原文。
- 本机私密照片属于例外：它们只能通过当前设备的浏览器 IndexedDB 展示，不得加入公开数据、同步包、源码或 Git 提交。

### 通过 Claude Code thread 直传照片（2026-07-24 起，用户约定）

用户会直接在这个 Claude Code 长期 thread 里发餐食照片，不必附文字。收到后无需再等同步包，直接执行「识别 → 写入 → 更新 context → 发布」全流程。规则：

- **张数决定顿数**：发 1 张 = 上一顿；发 2 张 = 上两顿。以此类推。
- **顺序即时序**：多张时第一张是更早的那顿，最后一张是更晚的那顿（例如 2 张 = 第一张午餐、第二张晚餐）。
- **顿次/日期推断**：结合当前日期和照片内容判断落到哪一天的早/午/晚餐。默认是当天最近吃完的餐；若按时序应回推到前一天（如清早只发一张，多半是昨天晚餐），就落到前一天。拿不准时按最合理的推断先写，并在回复里**明确写清记录到了哪天哪顿**，让用户一句话就能纠正。
- **食物识别沿用既有规则**：照片里每样食物都写下来（尤其主食、蛋、汤）；蛋写作 `蛋` 或 `煎蛋`（含 `蛋` 关键词，别写 `鸡鸡蛋`那类会被 `鸡` 误分的写法）；不认识分类的食物（如 `豆皮`）会归未分类、排餐次末尾、不计入本周种类数——照写即可，必要时在 context 里注明。
- **自动发布**：写好后按第 2 节/AGENTS 的发布流程更新 `dietRecords`、刷新本文件、提交，并发布（main + gh-pages）。用户已默认「直接 publish」，无需再确认。饮食数据变更会改到 `public/nutriflow.html`，gh-pages 会随之更新。
- 更新相关测试（`tests/rendered-html.test.mjs` 里饮食天数、本周食物种类数、`data-photo-owner` 计数等断言依赖固定当周数据，加了当周新的一天就要同步改）。

### Codex 更新任务

把同步包粘贴到指向本项目文件夹的新 Codex 任务，并写“按同步包更新 NutriFlow”。Codex 应：

1. 先读 `AGENTS.md`、本文件和本地私密上下文（若存在）。
2. 去重并更新采购、库存和饮食数据；不猜测模糊字段。
3. 更新界面、测试和离线缓存版本（若应用壳变化）。
4. 验证后同步更新本文件的当前状态与变更日志。
5. 提交并发布到实际使用的托管地址。

普通手机 ChatGPT 对话不能自动读取电脑本地仓库或本文件，因此两个对话之间必须通过同步包交接。新开的本地 Codex 任务只要工作目录仍是本项目，就能自动读取 `AGENTS.md` 并恢复上下文。

`tests/rendered-html.test.mjs` 除了对源码做正则断言，还会用 `node:vm` 加上一套最小 DOM 桩真正执行 `public/nutriflow.html` 里的脚本，并 `await renderShopping()`。这是为了捕获渲染期的运行时异常：`render()` 用 `void renderShopping()` 触发渲染，异常会变成静默的 rejected promise，浏览器控制台不报错、对应区块只是空白，纯源码断言查不出来。改动渲染逻辑后要确认这两个运行时测试仍然通过。
